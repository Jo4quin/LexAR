# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Branch policy

All work happens on the `Lucaspini01` branch. Do not commit or push to `main` unless the user explicitly
asks for it in that specific instance — a prior approval does not carry over to later requests. Before
committing, confirm the current branch is `Lucaspini01` (or another branch the user explicitly named);
if you're on `main`, switch to `Lucaspini01` first (creating it from `main` if it doesn't exist yet)
rather than committing there.

## What this is

LexAR is an academic NLP project (Procesamiento de Lenguaje Natural coursework). It started as a
searchable knowledge base over Argentine national legislation used to surface likely redundancies,
contradictions, or overlaps between laws (Fases 1-3, complete). On 2026-07-08 the team pivoted to a
**legal assistant product for lawyers** built on that same corpus: a law explorer with AI-generated
relationship summaries, CSJN case law from SAIJ (2020 onward), and a RAG chatbot. Fases 1-3 are not
discarded — they're the product's retrieval engine and classification signal. See `PLAN.md` for the
full phase-by-phase implementation plan of the pivot (Fases 4-8).

The full project brief, data sources, phased plan, and evaluation strategy live in `README.md` — read it
before making architectural changes; it is the source of truth for scope and is written in Spanish.

## Notebook architecture (Fases 1 & 2)

The notebook is a linear pipeline, driven by constants in the config cell near the top. `MAX_TEXT_VERSIONS`
and `MAX_FRAGMENTS_FOR_EMBEDDINGS` default to `None` (full corpus); set both to a small number (e.g. `200`)
for a cheap smoke test before a real run — Vertex AI calls cost money.

1. **Fase 1 — Consolidacion de datos**: loads `text_versions.parquet`, joins it with `documents.csv` on
   `document_id`, then picks one text version per document (`actualizado` when available, else whatever
   exists) so the same norma doesn't appear twice and generate trivial self-candidates.
2. **Segmentacion**: `segment_text_version()` looks for `ARTICULO <n>` headers via `ARTICLE_PATTERN`
   (a normalized, accent-stripped regex match). If 2+ article headers are found, the text is split by
   article, with long articles (> `MAX_FRAGMENT_CHARS`) further split into overlapping chunks via
   `chunk_text()`. If fewer than 2 article headers are found, the whole text falls back to
   `chunk_text()` directly. Every fragment gets a `content_hash` (sha256) and a `fragment_id`
   (`frag:%08d`). This is the `legal_fragments` table from the README's schema. On the full corpus this
   produces ~114k fragments (~112.5k unique by `content_hash`).
3. **Fase 2 — Embeddings**: real semantic embeddings via Vertex AI (`gemini-embedding-001`, truncated to
   768 dims via Matryoshka Representation Learning). Requests are batched (`EMBEDDING_BATCH_SIZE`) and
   fired concurrently (`EMBEDDING_MAX_WORKERS` threads) through an `AdaptiveRateLimiter` — Vertex AI's
   per-minute token quota isn't published per-project, so instead of guessing it the limiter starts fast
   and backs off (AIMD, like TCP congestion control) whenever it hits `429 RESOURCE_EXHAUSTED`, converging
   to the real ceiling automatically. A batch that exhausts its retries is dropped and left pending rather
   than crashing the run. Results are checkpointed incrementally to `outputs/embeddings/part_*.parquet`
   (new file every `CHECKPOINT_EVERY` batches) so an interrupted run resumes without re-paying for work
   already done — checkpoints are read back and reconciled one file at a time (not concatenated into one
   big in-memory table) to keep peak memory bounded. The old dependency-free hashing baseline
   (`hashing_embeddings()`, TF-IDF-like token hashing) is kept only for the Fase 2.3/2.8 sanity comparison,
   not used for retrieval.
4. **Retrieval**: `top_neighbors_faiss()` builds a `faiss.IndexFlatIP` over the L2-normalized embeddings
   (exact cosine similarity via inner product) and searches for `top_k * over_fetch` neighbors per
   fragment, then filters out self-matches and same-`document_id` pairs and keeps the top `TOP_K_NEIGHBORS`.
   **Memory note**: the candidates table does NOT include fragment text by default
   (`include_text=False`) — at full-corpus scale (~680k pairs) duplicating text_a/text_b per row is what
   caused repeated `MemoryError`s during development. Text is joined back in only for the top-500 export
   and for small sanity-check samples (see cell 34, `include_text=True`). If you see a `MemoryError` on
   `index.add()`/`pd.DataFrame(...)` while modifying this cell, this is almost certainly why — don't
   re-add text_a/text_b to the bulk table.
5. **Clusters exploratorios**: `connected_components_from_candidates()` does a simple BFS over an
   adjacency graph built from candidate pairs above a similarity threshold recalibrated as the 90th
   percentile of the observed score distribution (a fixed threshold tuned for the old hashing baseline's
   score range doesn't transfer to semantic-embedding scores).
6. **Visualization**: PCA and t-SNE (not UMAP — `umap-learn`'s optional `torch` backend triggers a native
   DLL conflict with `faiss`/`tensorflow` on Windows) over a 10k-fragment sample, colored by `tipo_norma`
   and by decade of `fecha_sancion`.

Last full run: 113,895 fragments → 112,582 unique embeddings → 679,720 candidate pairs → 8,942 exploratory
clusters (similarity threshold ≈0.957). Embedding cost was a few dollars against the UDESA GCP credits.

**Known fragility**: getting a truly full-corpus run to complete end-to-end inside one Jupyter kernel is
memory-sensitive on machines with limited free RAM (observed `MemoryError` even after the FAISS fix above,
resolved by freeing intermediate DataFrames — see the `del ...; gc.collect()` calls after cells 10, 13, and
17 — and by closing other memory-heavy applications before running). If it happens again, check
`outputs/embeddings/part_*.parquet` first — embeddings are checkpointed independently of the rest of the
pipeline and don't need to be regenerated.

**If cell outputs revert to stale/old content**: this happened once during development — an editor
(VS Code) had the notebook open from before a round of edits, and its autosave overwrote the file with its
stale in-memory version. Before re-running a long pipeline execution, make sure the notebook isn't open
in another editor, or close it without saving first.

**Text encoding note**: the source text in `text_versions.parquet` has mojibake in accented characters
(e.g. "Decl�rase" instead of "Declárase") inherited from the original Infoleg/SAIJ scrape. This surfaces
throughout every downstream table (fragments, embeddings, classifications) — it's cosmetic (doesn't affect
segmentation, embeddings, or LLM classification quality, all confirmed working correctly on this text) but
would need fixing at the source if exact-text quotes matter for the Fase 5 report/demo.

## Notebook architecture (Fase 3 — `Clasificacion_Candidatos.ipynb`)

Reads `outputs/analysis_candidates.parquet` and `outputs/embedding_fragments.parquet` from Fase 2.
`MAX_PAIRS_FOR_CLASSIFICATION` in the config cell gates a smoke test the same way `MAX_TEXT_VERSIONS` does
in Fase 1-2 — but for a smoke test, it takes a **random sample** (not `head()` on similarity), because the
highest-similarity pairs are almost all near-duplicates that the `near_identical` rule resolves without
ever reaching the LLM, which would leave a top-N smoke test with nothing to classify.

1. **Scope**: candidates are filtered to `similarity_score >= SIMILARITY_THRESHOLD` (0.957, the p90
   threshold from Fase 2) — 68,050 of 679,720 pairs. Text and `tipo_norma` for both fragments are joined in
   from `embedding_fragments.parquet`; `relations.csv` (Infoleg's document-level modification graph) is
   joined in as a `has_infoleg_relation` boolean, passed to the LLM as a hint, never as an auto-label.
2. **Rules** (no LLM cost, resolved 35% of the 68,050 pairs on the full run — 22,831 boilerplate + 1,058
   near-identical):
   - `rule:near_identical` (similarity >= 0.999 or identical normalized text) → `possible_overlap`.
   - `rule:boilerplate` → `neutral`, on **either** of two independent signals: (a) a closing-formula regex
     (`comuniquese al poder ejecutivo`, `dada en la sala de sesiones`, etc.) matching **both** fragments —
     dates/signers vary per law, so these rarely hit a repetition threshold and needed a text-only signal;
     (b) both fragments being high-degree nodes (`BOILERPLATE_MIN_DEGREE`, default 30) in the candidate
     graph — catches generic repeated clauses that don't match the closing-formula regex (e.g. standard
     procedural cross-references). Requiring the condition on **both** sides of the pair keeps it
     conservative. An earlier version required degree AND regex-match-on-one-side together, which fired on
     0 pairs in a 100-pair smoke sample — the AND was too strict for either signal alone.
3. **Classification engine**: same `AdaptiveRateLimiter` (AIMD) + `ThreadPoolExecutor` + resumable
   part-file checkpointing pattern as Fase 2's embeddings, generalized into one `run_classification_stage()`
   function reused for both LLM tiers below. One pair per LLM call, structured JSON output via
   `types.Schema`/`response_mime_type="application/json"` (label enum, confidence, explanation, evidence
   quotes from both fragments) — no manual JSON parsing/retry logic needed.
4. **Triage** (`gemini-2.5-flash-lite`): classifies every pair the rules didn't resolve (44,161 on the full
   run). Rate-limiter-bound throughput ramped from ~800/hr to ~14,000/hr as it converged on the project's
   real quota; the full triage pass took about 3.5 hours end-to-end with zero dropped/failed batches.
5. **Verification** (`gemini-2.5-flash`): re-classifies only the pairs triage labeled `possible_conflict`
   (440 on the full run) with the stronger model, since conflicts are the highest-stakes finding. This
   caught real triage over-eagerness: only 84 of 440 (19%) were confirmed as `possible_conflict` on
   verification — the rest were downgraded to `different_scope` (218), `possible_modification` (122),
   `possible_overlap` (13), or `needs_review` (3). `final_label` prefers verification > triage > rule.
6. **Consolidation & analysis**: `candidate_classifications.parquet` merges rule/triage/verify results;
   pairs sent to the LLM but left unresolved after exhausting retries are marked `needs_review` rather than
   silently dropped (0 of 397 `needs_review` on the full run were retry failures — all were genuine LLM
   decisions). `conflicts_top.csv` joins full text back in for the small `possible_conflict` set only (same
   memory-safety pattern as Fase 2's `analysis_candidates_top.csv` — never add text_a/text_b to the bulk
   table).
7. **Golden set**: `golden_set_para_etiquetar.csv` stratifies by `(final_label, similarity_band)` to
   guarantee coverage of rare labels like `possible_conflict`, with an empty `human_label` column for
   manual annotation. This is a deliberate selection-bias tradeoff (not a random corpus sample) — documented
   in the notebook and worth repeating in the Fase 5 report. The notebook's evaluation cell (confusion
   matrix, human-LLM agreement, Precision@K on `possible_conflict`) is a no-op until `human_label` is filled
   in by hand.

**Full-run sanity check**: `possible_modification` correlates strongly with `has_infoleg_relation` (60.6%
of `possible_modification` pairs vs. ~12-19% baseline in the overall candidate pool) — the LLM's
modification calls line up with Infoleg's independently-sourced modification graph, a good signal the
classification isn't noise. `possible_conflict` pairs have `has_infoleg_relation = False` in 100% of cases
(84/84) — genuine contradictions are between *unrelated* norms, not norms that already know about each
other via an official modification link.

Last full run (2026-07-08): 68,050 pairs in scope → 23,889 resolved by rules → 44,161 sent to triage → 440
flagged `possible_conflict` → 84 confirmed on verification. Final distribution: 25,017 `possible_overlap`,
24,868 `neutral`, 14,077 `possible_modification`, 3,607 `different_scope`, 397 `needs_review`, 84
`possible_conflict`.

### Fase 4 — Jurisprudencia CSJN (`notebooks/Jurisprudencia_CSJN.ipynb`, `src/lexar/saij.py`) — DONE

Scope agreed with the team on 2026-07-08: **only CSJN (Corte Suprema) rulings, 2020 onward**, not all
of SAIJ's jurisprudence. Final run (2026-07-09): **1,234 fallos** (100% `fetch_status=ok`), **8,702**
unique fragments, **8,702/8,702** embeddings, **28,930** law<->case links across 1,225 fallos and
2,743 laws.

- **API discovered by exploration** (same family of endpoints that produced `saij_texts.csv`):
  - Search: `https://www.saij.gob.ar/busqueda?o=<offset>&p=<page_size>&f=<facets>&s=fecha-rango|DESC`.
    **Gotcha**: the tribunal facet is `Tribunal/CORTE SUPREMA DE JUSTICIA DE LA NACION` — `Organismo/...`
    returns 0 results despite looking equally plausible.
  - Document metadata: `https://www.saij.gob.ar/view-document?guid=<uuid>`. The full text is **not** in
    this JSON — it's referenced as a PDF (`content['texto-doc']`).
  - PDF download: `https://www.saij.gob.ar/descarga-archivo?guid=<pdf_uuid>&name=<file>`. Extracted with
    `pypdf`; a handful of pages have no text layer (`fetch_status = 'pdf_no_text'`) and are excluded
    rather than crashing the run.
  - **Gotcha, central and non-obvious**: SAIJ's search pagination returns a deterministic HTTP 500 past
    offset ~1000 (reproduced repeatedly at `o=1000, p=100` — not transient, retrying never helps). The
    first fix attempted was `_search_page_resilient()` binary-splitting the page on a persistent 500 to
    isolate and skip individual bad offsets — this seemed to work in a first probe, but running it for
    real revealed the *entire* range past offset 1000 fails, not one corrupt document: it's a hard
    pagination-depth ceiling on SAIJ's backend (typical of Elasticsearch/Solr with a low
    `max_result_window`), not a data quirk. **The actual fix**: query one calendar year at a time via the
    `Fecha/<year>` facet (confirmed exact: `Fecha/2020` returns precisely 248 results, matching the known
    per-year facet breakdown). Since CSJN has at most ~250 fallos/year, every per-year query stays well
    under the 1000-offset ceiling. `search_fallo_uuids(since_year, until_year)` in `saij.py` implements
    this; the old offset-skipping bisection in `_search_page_resilient()` is still used *within* each
    year's pagination as a defense-in-depth fallback, not as the primary strategy.
  - **Gotcha**: SAIJ's `numero-interno` field comes back as either `int` or `str` depending on the fallo,
    which breaks `pyarrow`'s schema inference across checkpoint parts. Every scalar field is cast to
    `str` before writing (`_s()` helper in `saij.py`).
  - **Gotcha, found 2026-07-09 via the app's "ver en SAIJ" link (real bug, not cosmetic)**: `fetch_fallo()`
    built the public document URL as `{BASE_URL}/{friendly-url.description}` using only the SEO slug the
    API returns. SAIJ's site (Struts2/Tomcat) actually requires the document's uuid as a **second path
    segment**, `{BASE_URL}/{slug}/{uuid}` — without it, `DocumentDisplayAction.displayDisplay` throws a
    server-side `NullPointerException` (HTTP 500). Confirmed by navigating both URL forms against the
    live site and comparing to a working link found on saij.gob.ar's own homepage. Fixed in `saij.py`
    (`url` now includes `/{uuid}`); the 1,234 already-scraped rows in `fallos_csjn.parquet` were patched
    in place (slug was already correct, just needed the `/{uuid}` suffix appended) instead of
    re-scraping.
- Scraping is checkpointed like Fase 2 (part files, resumable) and polite (~0.4s between requests — no
  documented rate limit, unlike Vertex AI, so this is just good citizenship, not quota management).
- Segmentation reuses `chunk_text()` directly (fallos have no `ARTICULO n` structure).
- Embeddings reuse the exact Fase 2 pipeline (`embed_corpus_checkpointed()` / `consolidate_embeddings()`
  in `src/lexar/embeddings.py`) — same model, same `SEMANTIC_SIMILARITY` task type, same 768d.
- Law<->case links (`build_law_case_links()` in `links.py`): each case fragment searched against the
  Fase 2 FAISS law index, aggregated to `(document_id, case_id)`. Same memory rule as
  `analysis_candidates`: no `text_a`/`text_b` in the bulk table.

**Gotcha found finishing this fase, real bug not a resource issue**: while retrying the embedding run
after a disk-full incident (see below), a manually-deleted corrupted checkpoint part
(`part_000007.parquet`) left a gap in the numbering. `embed_corpus_checkpointed()`'s next-file-index
logic was `len(list(checkpoint_dir.glob("part_*.parquet")))` — a *count*, not a max index. With a gap,
that count collides with an existing higher-numbered file, and the next `to_parquet()` call **silently
overwrites** a valid checkpoint (parquet writes don't error on overwrite). This looked exactly like a
disk-space problem — the "missing embeddings" count flip-flopped instead of monotonically decreasing
across retries — until traced to this. Fixed in `embeddings.py` with `_next_part_index()`, which computes
`max(existing indices, default=-1) + 1`. **If a checkpoint file is ever deleted by hand for any reason,
this class of bug can recur wherever the same `len(glob(...))` pattern is used** — the original Fase 2
notebook cell 19 has the identical pattern, never fixed there since nobody deleted a checkpoint file
during that phase's development (the notebook is intentionally left untouched, per this file's own
"Fases 1-3 don't get touched" convention — this is a latent, dormant bug there, not misinformation).

### Fase 5 — Vinculos entre normas (`notebooks/Vinculos_Normas.ipynb`, `src/lexar/links.py`, `summaries.py`) — DONE

`build_norm_links()` aggregates Fase 2's fragment-level `analysis_candidates` and Fase 3's
`candidate_classifications` up to document level, outer-joined with Infoleg's official `relations.csv`
graph so a norm's official modifications show up even without a semantic match. Output:
`outputs/norm_links.parquet`, one row per `(document_id_a, document_id_b)` pair with `link_source`
(`semantic` / `official` / `both`), `dominant_label` (mode of `final_label`, excluding
`rule:boilerplate` votes), and up to 3 sample explanations.

**Result (2026-07-09)**: 48,276 links (38,552 semantic-only, 8,297 official-only, 1,427 both).
`dominant_label` distribution: 23,357 `neutral`, 11,575 `possible_modification`, 10,865
`possible_overlap`, 2,277 `different_scope`, 192 `needs_review`, 10 `possible_conflict`. Sanity check
against Ley 24.240 (Defensa del Consumidor, `infoleg:638`, a heavily-amended law): 20 links, 18
correctly detected as later official modifications.

**Gotcha**: `relations.csv` has ~115k of its 126k rows with an empty `source_document_id` or
`target_document_id` (relations whose other end falls outside the Ley/Decreto-Ley corpus scope) — must
read with `dtype=str, keep_default_na=False` and filter empties before joining, or a mixed-type column
breaks `np.minimum`/`np.maximum` used to build the undirected pair key.

**Team decision (2026-07-08)**: link summaries are **on-demand with a parquet cache**
(`outputs/link_summaries.parquet`, keyed by `doc_pair_key` + prompt version), not a Fase-3-style batch
job — generated the first time someone opens that link in the app. `LinkSummarizer` in `summaries.py`
handles both the generation and the cache read/write; the notebook only pre-warms a handful of demo
examples.

### Fase 6 — Chatbot RAG (`src/lexar/chatbot.py`) — DONE

`answer_case()`: (1) an LLM call rewrites the colloquial case into 1-3 formal legal-language queries
(`rewrite_query()` — this is why re-embedding the corpus with `RETRIEVAL_*` task types wasn't needed:
the query is embedded with the same `SEMANTIC_SIMILARITY` model as the corpus, a symmetric space); (2)
FAISS search over both the law index and the case index, aggregated to document/case level; (3) a
second LLM call drafts the answer with mandatory bracketed citations (`[fragment_id]`); (4) each
citation is checked automatically against the source fragment's normalized text
(`normalize_text()` from `segmentation.py`) — this backs the Fase 8 traceability metric and the app's
citation checkmarks.

**Result (2026-07-09)**: verified end-to-end against both live FAISS indices (112,582 law fragments +
8,702 case fragments). Reference example ("me chocaron el auto y el otro conductor no tiene seguro"):
rewrote to 2 legal-language queries, retrieved 6 relevant laws and 4 CSJN fallos, generated a cited
answer with 8/10 valid citations. Across the full 17-case eval set (Fase 8): 143 citations total, 93.7%
traceable. Minor known refinement opportunity, not fixed: the model occasionally emits multiple
fragment IDs inside one citation bracket (e.g. `[cfrag:X, cfrag:Y]`), which the current validation
logic treats as a single (invalid) source id instead of splitting it — likely responsible for some of
the ~6% untraceable citations.

### Fase 8 — Evaluation (`notebooks/Evaluacion_Producto.ipynb`) — DONE

Uses `eval/casos_prueba.csv` (17 hand-annotated colloquial cases with expected `numero_norma` values,
matched to `document_id` via `document_identifiers.csv`) to compute Precision@K/Recall@K for the
chatbot's retrieval, with an ablation comparing raw-query vs. rewritten-query retrieval (justifies the
query-rewriting design decision with a number instead of just intuition). Also computes the % of
chatbot citations that pass automatic quote validation, and produces a stratified sample
(`outputs/eval/rubrica_resumenes.csv`) for manual review of link-summary quality, following the same
stratified-sampling pattern as Fase 3's golden set (that rubric was generated but left unlabeled — it
needs a human pass, same as Fase 3's `human_label` column).

**Result (2026-07-09)**, saved to `outputs/eval/ablation_query_rewrite.csv` and
`outputs/eval/citation_traceability.csv`:
- Precision@10: 10.0% without rewriting vs. 9.4% with — essentially flat. Expected: each case has only
  1-2 expected laws, so precision@10 is structurally low regardless of retrieval quality.
- **Recall@10: 70.6% without rewriting vs. 73.5% with rewriting** — the number that actually validates
  the query-rewriting design decision, since for a research-assistant chatbot recall (does the right law
  show up anywhere in the list the user sees) matters more than precision.
- Citation traceability: 93.7% (134/143).

## Cross-cutting gotchas from the pivot

- **Low-memory FAISS loading**: this dev machine runs as low as ~0.6-1.5 GB free RAM (worse than the
  Fase 2 `MemoryError`s already documented above). `CorpusIndex` in `retrieval.py` opens the `.npy`
  embedding matrix with `mmap_mode="r"` and adds to the FAISS index in 20k-row chunks rather than
  materializing the whole matrix, and — more importantly — **never loads the `text` column into memory
  at all**: `EMBEDDING_FRAGMENTS_PATH`/`CASE_FRAGMENTS_PATH` are opened as a `pyarrow.dataset.Dataset`
  and `.take()` is used to fetch text for only the rows a given search actually returns. If you add a
  new bulk read of `embedding_fragments.parquet`, do not add `text` to `columns=` unless you've checked
  it's bounded (same spirit as the existing "no text_a/text_b in bulk tables" rule).
- **`ROOT` vs. `REPO_DIR`** in `config.py`: `ROOT` resolves to wherever `data/` and `outputs/` live
  (shared across worktrees — these are gitignored, so a worktree checkout doesn't have its own copy);
  `REPO_DIR` is always the actual git checkout the code lives in (`Path(__file__).resolve().parents[2]`),
  used only for versioned files like `eval/casos_prueba.csv`. Don't conflate the two — pointing
  `CASOS_PRUEBA_PATH` at `ROOT` would silently read a stale copy from a different worktree.
- **Disk-full during embedding runs writes silently-truncated parquet checkpoints**: during Fase 4's
  embedding run the C: drive filled to 0 bytes free. `to_parquet()` calls did not always raise an
  exception on the truncated writes — the process would print a success message and even report "0
  batches discarded," but `consolidate_embeddings()`'s post-hoc integrity check (missing-hash count)
  would still be nonzero, and would *not* monotonically decrease across retries. If you see that
  pattern (missing count going up as often as down across reruns), check `df -h` before assuming a code
  bug — this looked identical to the separate checkpoint-index-collision bug above until both were
  diagnosed in the same session; they compounded (the collision bug persisted after disk space was
  freed, since it wasn't disk-related at all).
