## Branch policy

All work happens on the `Lucaspini01` branch. Do not commit or push to `main` unless the user explicitly
asks for it in that specific instance — a prior approval does not carry over to later requests. Before
committing, confirm the current branch is `Lucaspini01` (or another branch the user explicitly named);
if you're on `main`, switch to `Lucaspini01` first (creating it from `main` if it doesn't exist yet)
rather than committing there.

## What this is

LexAR is an academic NLP project (Procesamiento de Lenguaje Natural coursework) that builds a searchable
knowledge base from Argentine national legislation and uses it to surface likely redundancies,
contradictions, or overlaps between laws. A planned second phase layers a legal-assistant use case
(drafting/reviewing legal writing with citations) on top of the same corpus — not yet started.

The full project brief, data sources, phased plan, and evaluation strategy live in `README.md` — read it
before making architectural changes; it is the source of truth for scope and is written in Spanish.

The notebook's config cell resolves `ROOT` to whichever of `Path.cwd()` or its parent contains a `data/`
directory, so it works whether Jupyter is launched from the repo root or from inside `notebooks/`. It then
asserts `TEXT_VERSIONS_PATH` and `DOCUMENTS_PATH` exist under `data/lexar_datos_infoleg_saij/` and will

Outputs are written to `outputs/` at the repo root (created automatically, gitignored):
- `outputs/legal_fragments.parquet` — all segmented fragments from Fase 1 (+ a 2,000-row `_sample.csv` for quick inspection).
- `outputs/embeddings/part_*.parquet` — incremental embedding checkpoints, keyed by `content_hash` (see below).
- `outputs/embeddings.npy` + `outputs/embedding_fragments.parquet` — consolidated, L2-normalized embedding matrix aligned 1:1 with its metadata.
- `outputs/analysis_candidates.parquet` — full candidate-pairs table from Fase 2 (no full text — see memory note below).
- `outputs/analysis_candidates_top.csv` — top 500 pairs by similarity, with text joined back in, for manual review.

## Notebook architecture (Fases 1 & 2)

The notebook is a linear pipeline, driven by constants in the config cell near the top. `MAX_TEXT_VERSIONS`
and `MAX_FRAGMENTS_FOR_EMBEDDINGS` default to `None` (full corpus); set both to a small number (e.g. `200`)
for a cheap smoke test before a real run — Vertex AI calls cost money.

1. **Fase 1 — Consolidacion de datos**: loads `text_versions.parquet`, joins it with `documents.csv` on
   `document_id`, then picks one text version per document (`actualizado` when available, else whatever
   exists) so the same norma doesn't appear twice and generate trivial self-candidates.
2. **Segmentacion**: `segment_text_version()` looks for `ARTICULO <n>` headers via `ARTICLE_PATTERN`
   (a normalized, accent-stripped regex match). If 2+ article headers are found, the text is split by
   article, with long articles (> `MAX_FRAGMENT_CHARS`) further split into overlapping chunks via
   `chunk_text()`. If fewer than 2 article headers are found, the whole text falls back to
   `chunk_text()` directly. Every fragment gets a `content_hash` (sha256) and a `fragment_id`
   (`frag:%08d`). This is the `legal_fragments` table from the README's schema. On the full corpus this
   produces ~114k fragments (~112.5k unique by `content_hash`).
3. **Fase 2 — Embeddings**: real semantic embeddings via Vertex AI (`gemini-embedding-001`, truncated to
   768 dims via Matryoshka Representation Learning). Requests are batched (`EMBEDDING_BATCH_SIZE`) and
   fired concurrently (`EMBEDDING_MAX_WORKERS` threads) through an `AdaptiveRateLimiter` — Vertex AI's
   per-minute token quota isn't published per-project, so instead of guessing it the limiter starts fast
   and backs off (AIMD, like TCP congestion control) whenever it hits `429 RESOURCE_EXHAUSTED`, converging
   to the real ceiling automatically. A batch that exhausts its retries is dropped and left pending rather
   than crashing the run. Results are checkpointed incrementally to `outputs/embeddings/part_*.parquet`
   (new file every `CHECKPOINT_EVERY` batches) so an interrupted run resumes without re-paying for work
   already done — checkpoints are read back and reconciled one file at a time (not concatenated into one
   big in-memory table) to keep peak memory bounded. The old dependency-free hashing baseline
   (`hashing_embeddings()`, TF-IDF-like token hashing) is kept only for the Fase 2.3/2.8 sanity comparison,
   not used for retrieval.
4. **Retrieval**: `top_neighbors_faiss()` builds a `faiss.IndexFlatIP` over the L2-normalized embeddings
   (exact cosine similarity via inner product) and searches for `top_k * over_fetch` neighbors per
   fragment, then filters out self-matches and same-`document_id` pairs and keeps the top `TOP_K_NEIGHBORS`.
   **Memory note**: the candidates table does NOT include fragment text by default
   (`include_text=False`) — at full-corpus scale (~680k pairs) duplicating text_a/text_b per row is what
   caused repeated `MemoryError`s during development. Text is joined back in only for the top-500 export
   and for small sanity-check samples (see cell 34, `include_text=True`). If you see a `MemoryError` on
   `index.add()`/`pd.DataFrame(...)` while modifying this cell, this is almost certainly why — don't
   re-add text_a/text_b to the bulk table.
5. **Clusters exploratorios**: `connected_components_from_candidates()` does a simple BFS over an
   adjacency graph built from candidate pairs above a similarity threshold recalibrated as the 90th
   percentile of the observed score distribution (a fixed threshold tuned for the old hashing baseline's
   score range doesn't transfer to semantic-embedding scores).
6. **Visualization**: PCA and t-SNE (not UMAP — `umap-learn`'s optional `torch` backend triggers a native
   DLL conflict with `faiss`/`tensorflow` on Windows) over a 10k-fragment sample, colored by `tipo_norma`
   and by decade of `fecha_sancion`.

Last full run: 113,895 fragments → 112,582 unique embeddings → 679,720 candidate pairs → 8,942 exploratory
clusters (similarity threshold ≈0.957). Embedding cost was a few dollars against the UDESA GCP credits.

**Known fragility**: getting a truly full-corpus run to complete end-to-end inside one Jupyter kernel is
memory-sensitive on machines with limited free RAM (observed `MemoryError` even after the FAISS fix above,
resolved by freeing intermediate DataFrames — see the `del ...; gc.collect()` calls after cells 10, 13, and
17 — and by closing other memory-heavy applications before running). If it happens again, check
`outputs/embeddings/part_*.parquet` first — embeddings are checkpointed independently of the rest of the
pipeline and don't need to be regenerated.

**If cell outputs revert to stale/old content**: this happened once during development — an editor
(VS Code) had the notebook open from before a round of edits, and its autosave overwrote the file with its
stale in-memory version. Before re-running a long pipeline execution, make sure the notebook isn't open
in another editor, or close it without saving first.

## Where this is headed (not yet implemented)

Per the README's phased plan, the next phases build on `outputs/analysis_candidates.parquet`:

- **Fase 3**: classify each candidate pair with an LLM/rules into one of
  `possible_conflict | possible_overlap | possible_modification | different_scope | neutral | needs_review`,
  storing label, confidence, explanation, cited fragments, model, and prompt version.
- **Fase 4**: for conflict/overlap cases, generate an alternative redaction proposal that cites the
  fragments it's based on.
- **Fase 5**: report and demo.

When implementing these, follow the field/table names already specified in the README rather than
inventing new ones, since Fases 1-2 already establish `fragment_id`, `document_id`, `candidate_id`,
`pair_key`, etc. as the shared keys new code should join against.
